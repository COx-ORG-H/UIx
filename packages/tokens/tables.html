Tables & data grids

The table is the workhorse surface of ITSM and CRM — queues, CMDB, contacts, audit. This chapter specifies one universal table system for UIx v2 by demonstration: every pattern below is live, themed, and keyboard-operable. The core is product-neutral; the ITSM / CRM appendix layers on masking, audit and regulatory behaviour.

One primitive, many surfaces. Incident queues, CI lists, contact tables and audit logs are the same component configured differently — not bespoke tables. Consistency here compounds across every screen and every product built on UIx.
Decide before you build

Five questions settle most table decisions up front. Answer them per surface — the answers drive density, default columns, sort, and which interactions you turn on.

QuestionWhy it changes the design
What's in the cells?Short codes & numbers tolerate high density; free text and relations need truncation, subtext, or a peek.
Scan or act?Info-seeking tables optimise comparison & filtering; action tables optimise selection, inline edit & bulk ops.
Which devices?Desktop-dense vs. phone-first changes the responsive ladder (see Responsive).
Editable?Read-only is simplest; editable adds the inline-edit ladder & confirmation feedback.
Customisable view?Power users need density, column show/hide/reorder, saved views — all persisted.
Three table archetypes
👟 Action-oriented
Linked into a workflow. Optimise selection, inline edit, bulk actions, fast row → detail. e.g. an agent's incident queue.
👓 Info-oriented
Large datasets, rich seeking. Optimise sort, filter, column control, scannability. e.g. CMDB CI explorer.
👟👓 Action + Info
Super-user surface needing depth on both. The full feature set — used sparingly. e.g. change management board.

Anatomy

The parts every table is assembled from. The container frames the data; the toolbar holds view-level controls; the sticky header labels and sorts; rows are the records; the footer carries count and paging.

1
Toolbar — search, filters, view menu, saved views. View-level controls live here, never in the header.
2
Select-all + bulk bar — header checkbox toggles all; selecting reveals a contextual action bar.
3
Column header — label aligned to its column; click to sort; drag the right edge to resize.
4
Row — one record. Hover reveals row actions; click the ID to peek.
5
Cell — typed: text, number, status pill, user, date, relation.
6
Footer — result count + pagination / load-more. Stays put while the body scrolls.

Cell types & alignment

Alignment is not cosmetic — it's how the eye scans and compares. Get these defaults right and most tables read well with no further tuning.

Do
Left-align text & IDs. Right-align quantitative numbers with tabular-nums so digits line up by place value. Match each header's alignment to its column.
Don't
Never centre-align data — it breaks the scan line and hides irregularities. Don't repeat the column word in every cell (“Qualified Lead”, “New Lead”). Don't proportional-space figures.

Row styles & dividers

Four ways to separate records. Pick one per product and hold to it. With interactivity (hover, selected, disabled, pinned), zebra striping stacks too many greys — prefer line dividers for dense, stateful tables.

The five-greys problem. Zebra + hover + selected + disabled + pinned can put five different background greys in play at once, breaking visual continuity. If your table is interactive and stateful, drop zebra and let the 1px divider do the work.

Sorting

A quick win with high returns. Click a header to cycle ascending → descending → none. The active column shows a solid arrow in the accent colour; idle sortable columns show a faint hint on hover. The indicator never disturbs the header's alignment to its column.

Choose a sensible default

Default sort should answer “what do I look at first?”. Usually that's most recent (newest created/updated) or most needing action (highest priority, soonest SLA breach). ITSM queues default to priority ascending, then opened descending.

Multi-column sort

Hold ⇧ and click to add a secondary sort; a small ordinal (1, 2) appears on each sorted header. Keep it optional — most users never need it, but power users lean on it hard.

Selection & bulk actions

Selection turns a list into a workbench. The header checkbox selects the page (indeterminate when partial); selecting any row reveals a contextual action bar. Actions only appear once they're relevant — a deliberate use of space that keeps the resting table clean.

Select-all means “this page”. When a filter or large dataset means more rows exist off-page, offer a follow-up: “All 9 selected · Select all 418 matching”. Never silently act on rows the user can't see.

Bulk actions are destructive-aware: Delete confirms and is styled danger; reversible actions (assign, tag, export) act immediately with an undo toast. Pin (📌 on hover) lifts a row to the top and survives sort & filter.

Column management

Give users control over what they see and they engage more and tire less. The consolidated View menu holds density, zebra, freeze and per-column show / hide / reorder. Drag a header's right edge to resize. All of it persists per user per surface, with one reset.

When there are too many columns
Reach for first
Fix the default. Decide as a team which 5–7 columns matter on load. A good default beats any amount of customisation. Hide the rest behind the View menu.
Then layer
Freeze the identifier column & horizontal-scroll the rest · let users add/remove & resize · always provide Reset to default.
Freeze the identifier. On horizontal scroll, the leftmost column (ID / name) stays sticky — the fixed point that keeps every other value legible. Toggle it in the View menu; try resizing then scrolling above.

Search, filter & saved views

The toolbar centralises every way to narrow the data: full-text search with match highlighting, typed filter chips, the View menu, and saved views. Search is reachable with Mod/; Esc clears it.

Views are URLs. Filters, sort, visible columns and density serialise into the query string, so a “DORA-relevant, unresolved, oldest-first” view is a link a colleague can open to the exact same render. Saved views are named bundles of that state, scoped to a user or shared with a team.

Pagination & loading large sets

Three strategies, each with a clear home. The deciding factor is whether users need to return to a known position (pagination) or browse a continuous stream (load-more), and how big the set gets (virtualization).

Numbered pagination

Bounded sets where position matters — “page 3 of 12”. Predictable, linkable, server-friendly.

Load more

Exploratory browsing of a stream. Smooth, but harder to return to a spot and total can feel unknown — so show the count.

StrategyBest forWatch out for
PaginationReference data, audit, anything you cite by position; server-side data.Interrupts flow; extra clicks to scan.
Load more / infiniteFeeds, activity, casual browsing.No stable position; show running count & a footer once exhausted.
VirtualizationLong client-held lists (≥100 rows): render only visible rows, keep DOM bounded.Needs fixed/estimated row height per density; combine with either control above.
Virtualization is orthogonal. It's a rendering optimisation, not a UX pattern — the user still sees pagination or load-more. UIx virtualizes automatically past ~100 rows using the active density's row height as the estimate.

Row detail & editing

Tables are dense, so reveal interactions on demand rather than crowding the resting state. Hover a row for actions and the kebab menu; click the chevron to expand inline; click the ID to peek; double-click an editable cell to edit in place. Discoverability through hover is the through-line.

Click anywhere on a row — or focus it and press Enter (↑↓ move between rows) — to open the side peek. The checkbox, inline-edit cells, chevron and ⋮ menu are click-stoppers: they do their own thing and never trigger the peek, so the pointer cursor on the row is honest. In the peek, ↑↓ walk records and Esc closes. Chevron → inline expansion · double-click Priority or Assignee → inline edit (Enter commits with a ✓).

Peek is triage, not the destination. The whole row opens the peek for fast review, but the peek's Open full record must route to the entity's own shareable URL (e.g. /incidents/INC-2041) for deep work, multi-tab context and linking — a row that can't reach its record is a dead end, and a record with no URL breaks the shareable-link invariant. In this prototype that button is a stub (a styleguide has no record pages); click it in the peek to see the route it would take.
Showing more detail — the ladder
TooltipA peek at one hidden value; keeps full table context.

ModalFocused, but disrupts context. Use for short, self-contained detail.

Editing — match friction to stakes
Inline cellLowest friction, most context. For low-stakes, frequent edits (assignee, priority, tags).
Expandable / peek formModerate friction; room for validation. For multi-field edits.
Modal formDeliberate friction for high-stakes or destructive changes.
ConfirmationAlways close the loop: ✓ on commit, undo toast for reversible bulk ops, explicit Save where edits batch.

States

A table is rarely just “full of data”. Design the loading, empty, no-results and error states with the same care — they're where users spend their anxious moments. Each is token-driven and offers a clear next action.

Preview state

Loading uses skeleton rows shaped like the real ones (not a spinner) so layout doesn't jump. Empty (no records yet) invites the first action; No results (filters too tight) offers to clear them — a different message from empty. Error states stay calm, explain, and offer retry.

Responsive / multi-device

A wide table can't simply shrink. UIx steps the same data through a ladder as width drops — hide low-priority columns, freeze the identifier and scroll, then transform rows into cards. Resize the frame below (or your browser) and watch it adapt in real time; the layout reacts to actual available width, not a guessed breakpoint.

Frame
The responsive ladder
WidthPatternWhat happens
Desktop (≥ ~880px)Full tableAll columns; sticky header; hover actions.
Tablet (~560–880px)Priority columns + freeze & scrolldata-priority="low" columns drop; identifier freezes; the rest scroll horizontally.
Phone (< ~560px)Card transformEach row becomes a card — identifier as title, remaining fields as label : value. No horizontal scroll.
The anti-pattern: shipping the desktop table to a phone with a horizontal scrollbar. It forces users to swipe back and forth to read one record, carrying high cognitive load. If a complex table truly can't reduce, that's a signal to rethink what the phone user actually needs — not to scroll-cram it.

Accessibility & keyboard

A data table is only “done” when it's operable without a mouse and legible to assistive tech. UIx tables build on semantic <table> markup and add ARIA only where interaction demands it.

Keyboard map
TabMove between interactive elements (toolbar, header, row actions).
↑↓Move the active row; in a peek, walk between records.
SpaceToggle row selection; in the queue, open the peek.
EnterOpen the focused row's detail; commit an inline edit.
Mod/Focus the toolbar search.
EscClear search · cancel an edit · close peek/modal.
EnterOn a header: toggle sort. ⇧+ click adds multi-sort.
⌘ASelect all on page (when focus is in the grid body).
ARIA & semantics contract
ConcernImplementation
StructureReal <table><thead><tbody>. Use role="grid" only when cells are focusable/editable (a true data grid), with roving tabindex.
Sortaria-sort="ascending|descending" on the active th; the arrow is decorative (aria-hidden).
Selectionaria-selected="true" on the tr; header checkbox exposes indeterminate.
Row actionsIcon buttons carry aria-label; the kebab is a labelled menu button with a real menu.
Live updatesResult counts & async loads announce via a polite aria-live region.
FocusVisible :focus-visible ring (--uix-ring); focus is trapped in peek/modal and restored on close.
Motion & contrastAll transitions honour prefers-reduced-motion; status is never colour-only — pills pair a dot/label with the hue. Token pairs meet WCAG AA.
